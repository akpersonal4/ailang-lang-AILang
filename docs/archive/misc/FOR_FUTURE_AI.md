# For Future AI

A 10-minute guide to understanding the AILang project.

---

## Project Philosophy

AILang is an **AI-first, deterministic, specification-driven** language. It is designed to be reliably generated by large language models. The governing principles are:

- **Deterministic** — same input always produces same output
- **Explicit** — no hidden state, no magic, no implicit behavior
- **Specification-first** — `docs/reference/LANGUAGE_SPEC.md` is the single source of truth
- **AI-friendly** — unambiguous grammar, explicit braces, small keyword set
- **Tested** — TDD mandatory, 624+ tests
- **Minimal** — no features without evidence of need

Read `docs/governance/PROJECT_PHILOSOPHY.md` for the full manifesto.

## Architecture Overview

```
Source (.ail) → Lexer → tokens → Parser → AST → Semantic Analyzer → IR → Runtime → output
```

- **Compiler:** 40 Python source files, ~4,000 LOC. Single-pass, no multi-threading.
- **Runtime:** Tree-walking interpreter with lexical scoping. No bytecode, no JIT.
- **Stdlib:** 16 AILang modules. See `docs/reference/STDLIB_REFERENCE.md`.
- **Tests:** 624 tests across 28 test files. Pytest.
- **Applications:** 66+ across `apps/`, `ai_benchmarks/`, `examples/patterns/`.

## Language Constraints (Hard Rules)

| Constraint | ADR | Reason |
|-----------|:---:|--------|
| No loops — recursion only | ADR-001/002 | Simpler compiler, no loop semantics |
| No short-circuit `&&` — eager only | ADR-003 | Predictable execution, explicit `if` |
| No forward references — bottom-up order | ADR-004 | Single-pass compilation |
| Static lexical scoping | ADR-005 | Supports optimizations, intuitive |
| `let` needs initializer | §5.8 | No implicit undefined state |
| `return` needs value | §5.7 | Explicit always |
| `import` at top level only | §3.5 | Module resolution simplicity |
| `string.concat` takes exactly 2 args | §5.4 | Forces explicit intermediate steps |
| Map keys case-sensitive, opaque | §5.5 | No compile-time key checking |

## Runtime Constraints

- **No concurrency** — single-threaded, no threading support
- **No bytecode** — tree-walking interpreter
- **No JIT** — no dynamic compilation
- **Runtime frozen** — no further architecture changes after v0.2.0
- **Variable lookup cache** is the only optimization (RTO-001) — see `docs/runtime/optimizations.md`
- **Cache stores binding location, not value** — no invalidation for `assign`
- **Negative caching removed** — `assign` can create bindings in ancestors

## Common Mistakes (Pre-Compile)

See `AGENTS.md` §5 for the full list. Top 5:

1. **Forward reference** — `Undefined identifier: X` → move X before its caller
2. **Missing stdlib** — `split`, `find`, `join`, `sort`, `list.copy` don't exist → write custom
3. **Wrong map key** — keys mismatch between `set` and `get` → audit key names
4. **Variable collision** — `let i = 0` in multiple functions → use unique names
5. **Eager `&&`** — right operand executes even if left is false → use nested `if`

## Performance Philosophy

- **No optimization without profiler evidence** — ADR-007 mandates this
- **Pre-optimization complexity is waste** — runtime is frozen, no speculative tuning
- **The only bottleneck ever found was `Environment.resolve`** — at 85.4% of static analyzer runtime
- **If you believe there is a new bottleneck**, run `tools/python_profiler.py` first. Do not optimize without data.

## Before Changing Runtime Code

If modifying `compiler/runtime/`, `compiler/interpreter/`, `environment.py`, the semantic analyzer, or scope handling, read these first:

1. `docs/architecture/ARCHITECTURE_DECISIONS.md` — ADR-005 (lexical scoping), ADR-006 (lookup cache)
2. `docs/runtime/optimizations.md` — RTO-001 with all evidence
3. `docs/runtime/lookup_cache/design.md` — Why the cache works, what would break it
4. `docs/runtime/lookup_cache/implementation.md` — Exactly how the cache is implemented

**Never remove or redesign an optimization without understanding the architectural decision that introduced it.**
